SECTION 4: ADAPTIVE LEARNING (Phase D Final Feature)
=============================================================================

Overview:
  When a user marks 3+ consecutive notifications from the same tracking plan
  as "not_useful", the system automatically regenerates the search strategy
  using Gemini and notifies the user.

  Data flow:
    User clicks 👎  →  POST /api/notifications/{id}/feedback
      →  burilar_api.py counts consecutive not_useful for that plan
      →  if count >= 3: architect.refine_from_feedback(plan)
      →  updated strategy saved to plan  →  system notification added
      →  response includes { "adapted": true }
      →  frontend shows toast "追跡戦略が更新されました"

  Key constraint:
    notifier.py has NO reference to tracking_plans.
    All adaptation logic must live in burilar_api.py's feedback endpoint,
    which has access to both tracker.notifier and tracker.tracking_plans.

TASK D1.3 - Wire Adaptation in Feedback Endpoint (burilar_api.py)
-----------------------------------------------------------------------------
Status: COMPLETE
Effort: Medium (~1 hour)
File: burilar_api.py
Depends on: D1.1, D1.2

Update: POST /api/notifications/{notification_id}/feedback

Current code (lines ~289-301):
  success = tracker.notifier.submit_feedback(notification_id, feedback)
  if success:
      return jsonify({"status": "success"})

New logic after submit_feedback():

  1. Find the notification to get its plan_id:
       notification = next(
           (n for n in tracker.notifier.notifications
            if n["id"] == notification_id), None
       )
       plan_id = notification.get("plan_id") if notification else None

  2. If plan_id exists and plan_id in tracker.tracking_plans:
       plan = tracker.tracking_plans[plan_id]

       if feedback == "useful":
           plan.consecutive_not_useful = 0
           tracker.save_tracking_plans()

       elif feedback == "not_useful":
           plan.consecutive_not_useful += 1
           tracker.save_tracking_plans()

           if plan.consecutive_not_useful >= 3:
               try:
                   updated = tracker.architect.refine_from_feedback(plan)
                   # Apply updates
                   plan.strategy.update({
                       'search_queries': updated.get('search_queries', []),
                       'structure_items': updated.get('structure_items', []),
                       'missing_points': updated.get('missing_points', []),
                       'notification_triggers': updated.get('notification_triggers', [])
                   })
                   if updated.get('keywords'):
                       plan.keywords = updated['keywords']
                   plan.consecutive_not_useful = 0
                   tracker.save_tracking_plans()

                   # Add system notification
                   tracker.notifier.add_notification(
                       title="追跡戦略が更新されました",
                       message=updated.get('change_summary',
                           "フィードバックに基づき、より関連性の高い情報を追跡するよう戦略を更新しました。"),
                       type="info",
                       plan_id=plan_id
                   )

                   return jsonify({
                       "status": "success",
                       "adapted": True,
                       "message": updated.get('change_summary', '戦略を更新しました')
                   })
               except Exception as e:
                   print(f"[adaptive] Refinement failed: {e}")
                   # Fall through to standard success response

  3. Return standard: jsonify({"status": "success", "adapted": False})

Note:
  tracker.architect is not currently attached to tracker.
  Check if tracker has an architect attribute. If not, import and instantiate
  TrackingArchitect inside the except block or at the top of burilar_api.py.
  (See TASK D1.3a below)

SCALE REALITY CHECK
-----------------------------------------------------------------------------

  At 10,000 active users with continuous tracking:

  Load estimate:
    10,000 users × 2 avg plans = 20,000 tracking plans
    20,000 plans / 12hr avg frequency = ~1,700 Gemini calls/hour
    = ~28 Gemini calls/minute running concurrently

  Current bottleneck:
    background_checker() is a SINGLE Python thread checking all plans
    sequentially. At 20,000 plans × 5 sec/execution = 27 hours per cycle.
    Completely breaks at scale.

  Solution: Celery job queue (Phase E2 — add when ~500 active users)

  Gemini API cost at scale:
    ~1,700 calls/hour × $0.01-0.02/call = ~$400-800/month at 10K users
    Caching identical queries in Redis cuts this by 30-50% (Phase E4)

SECTION 7: DOMAIN MIGRATION GUIDE (burilar.com)
=============================================================================

Goal: Move burilar.com from current host → point to Vercel (frontend)
      and api.burilar.com → Railway (backend)

IMPORTANT: DNS changes take 0–48 hours to propagate globally (usually
under 1 hour with Cloudflare). Your old site stays live during propagation.
Do this during low traffic hours.

-----------------------------------------------------------------------------
STEP 1 — Find your domain registrar
-----------------------------------------------------------------------------

  Your domain is registered somewhere (GoDaddy, Namecheap, Google Domains,
  Porkbun, etc.). Log in to that registrar's dashboard.

  If you don't know which registrar:
    Go to https://lookup.icann.org and search burilar.com
    Look for "Registrar" in the results

-----------------------------------------------------------------------------
STEP 2 — Add burilar.com to Cloudflare
-----------------------------------------------------------------------------

  1. Go to cloudflare.com → Sign up (free) or log in
  2. Add a site → enter: burilar.com
  3. Select Free plan
  4. Cloudflare scans your existing DNS records — REVIEW THEM CAREFULLY
     Make sure any records for your current site are still there before
     changing nameservers (Cloudflare usually imports them automatically)
  5. Cloudflare gives you 2 nameservers, e.g.:
       ada.ns.cloudflare.com
       bob.ns.cloudflare.com
  6. DO NOT change nameservers yet — complete Step 3 first

-----------------------------------------------------------------------------
STEP 3 — Set up DNS records in Cloudflare BEFORE switching
-----------------------------------------------------------------------------

  While Cloudflare imported your old records, now ADD the new ones:

  For Vercel (main site — burilar.com):
    Type: A       Name: @    Value: 76.76.21.21      Proxy: ON
    Type: CNAME   Name: www  Value: cname.vercel-dns.com  Proxy: ON

  For Railway (API — api.burilar.com):
    Type: CNAME   Name: api  Value: <your-app>.up.railway.app  Proxy: ON
    (Railway gives you this CNAME when you add a custom domain)

  Keep any records for your OLD site until you confirm the new site works.
  You can delete old records after verifying burilar.com loads correctly.

-----------------------------------------------------------------------------
STEP 4 — Switch nameservers at your registrar
-----------------------------------------------------------------------------

  Go back to your domain registrar → find "Nameservers" or "DNS" settings
  Replace current nameservers with Cloudflare's two nameservers
  Save changes

  Registrar-specific instructions:
    GoDaddy:    My Products → DNS → Nameservers → Enter my own nameservers
    Namecheap:  Domain List → Manage → Nameservers → Custom DNS
    Google Domains / Squarespace: DNS → Use custom name servers
    Porkbun:    Domain Management → Authoritative Nameservers

-----------------------------------------------------------------------------
STEP 5 — Verify in Cloudflare
-----------------------------------------------------------------------------

  Cloudflare dashboard will show "Active" once nameservers are confirmed
  Usually takes 5–30 minutes after registrar change

  Test:
    curl -I https://burilar.com           → should serve Vercel frontend
    curl https://api.burilar.com/api/notifications  → should reach Railway

-----------------------------------------------------------------------------
STEP 6 — Disconnect from old host (after confirming new site works)
-----------------------------------------------------------------------------

  Only do this AFTER you've confirmed burilar.com loads your new app.

  What to do at old host:
    - Delete or suspend the old site/service there
    - Cancel the plan if you were paying for it
    - Remove old DNS records from Cloudflare (A/CNAME pointing to old host)

  What NOT to do:
    - Don't delete DNS records at old host before Cloudflare is active
    - Don't cancel old host plan until new site is confirmed working
